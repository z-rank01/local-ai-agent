# Plan: Local AI-Agent TUI 重构

## TL;DR

将 Open WebUI 替换为自建 Textual TUI 终端界面，gateway 逻辑嵌入 TUI 进程直连 Ollama，Docker 仅保留 skill 沙箱服务。对话存储用 SQLite，记忆系统统一到 workspace，移除 Tailscale 远程访问。

---

## 架构变更总览

```
【当前】                               【重构后】
Open WebUI (browser, :8888)           TUI (本地 Python 进程, Textual)
    ↓ OpenAI API                          ├── core/ (嵌入式 gateway 逻辑)
agent-gateway (Docker, :8700)             │   ├── Ollama (localhost:11434)
    ├── Ollama (Docker bridge)            │   ├── SQLite 对话存储
    ├── skill-files (Docker, :8100)       │   └── 统一记忆管理
    ├── skill-runner (Docker, :8200)      └── skill services (Docker)
    └── skill-websearch (Docker, :8300)       ├── skill-files (localhost:8100)
                                              ├── skill-runner (localhost:8200)
                                              └── skill-websearch (localhost:8300, 可选)
```

## Phase 1: Core 库提取 (从 gateway/ 重构)

> 将 gateway/ 的纯业务逻辑提取为可直接 import 的 Python 包，去除 FastAPI/Docker 依赖。

### 步骤

1.1 **创建 `core/` 包结构**
- 新建 `core/__init__.py`
- 新建 `core/config.py` — 集中配置管理，替代散落的 `os.environ.get()` 调用
  - 路径：`TOOLS_DIR`, `LOG_PATH`, `PROMPTS_DIR` 改为相对项目根目录
  - URL：`OLLAMA_BASE_URL` 默认改为 `http://localhost:11434`
  - URL：`SKILL_FILES_URL` / `SKILL_RUNNER_URL` / `SKILL_WEBSEARCH_URL` 默认改为 `http://localhost:8100/8200/8300`

1.2 **迁移纯逻辑模块** (*parallel*)
- `gateway/llm_client.py` → `core/llm_client.py` (无改动，仅更新 import)
- `gateway/tool_registry.py` → `core/tool_registry.py` (路径参数化)
- `gateway/tool_router.py` → `core/tool_router.py` (URL 参数化)
- `gateway/policy_engine.py` → `core/policy_engine.py` (路径参数化)
- `gateway/context_manager.py` → `core/context_manager.py` (无改动)
- `gateway/audit_logger.py` → `core/audit_logger.py` (路径参数化)
- `gateway/prompt_builder.py` → `core/prompt_builder.py` (路径参数化)

1.3 **提取 agentic loop 为独立模块** — `core/agent.py`
- 从 `gateway/app.py` 提取以下函数（已是纯逻辑，无 FastAPI 依赖）：
  - `_run_agent_loop()` → `Agent.run()`
  - `_run_tool_rounds()` → `Agent.run_tool_rounds()`
  - `_execute_tools_with_status()` → `Agent.execute_tools()`
  - `_prefetch_file_context()` → `Agent.prefetch_files()`
  - `_inject_context_into_messages()` → `Agent.inject_context()`
  - `_sse_stream()` 流式逻辑 → `Agent.stream()` async generator
- 将 `app.state.*` 全局变量改为 `Agent.__init__()` 的依赖注入参数
- 保留 `_TOOL_BUDGETS` / `_MAX_ROUNDS` 常量
- **关键**：保留 `strip_think_tags_from_history()` 调用（repo memory 已验证的关键步骤）

1.4 **统一记忆管理** — `core/memory_manager.py`
- 合并 `gateway/conversation_memory.py` + `gateway/memory_writeback.py` + `gateway/workspace_context.py`
- 去除 `SessionStore.fingerprint()` 指纹机制（TUI 有原生 conversation ID）
- 去除 `resolve_conversation_key()` 中的 HTTP header 解析（不再需要从 Open WebUI 推测 ID）
- 保留：MEMORY.md 读写、对话记忆读写、workspace 目录概览、记忆写回 LLM 调用
- 新增：用户偏好也存入 MEMORY.md（不再区分"Open WebUI Memory"和"workspace memory"）

1.5 **删除不再需要的模块**
- `gateway/session_store.py` — 指纹会话机制被 SQLite 替代
- `gateway/conversation_memory.py` — 合并入 memory_manager
- `gateway/memory_writeback.py` — 合并入 memory_manager
- `gateway/workspace_context.py` — 合并入 memory_manager
- `gateway/app.py` — FastAPI 入口，逻辑已提取到 core/agent.py
- `gateway/Dockerfile` — gateway 不再是 Docker 服务

**依赖关系**：1.1 → 1.2 (parallel) → 1.3 → 1.4 → 1.5

## Phase 2: 对话存储 (SQLite)

> 新建持久化对话存储，替代 Open WebUI 的对话管理。

### 步骤

2.1 **创建 `core/conversation_store.py`**

SQLite schema:
```sql
conversations(id TEXT PK, title TEXT, model TEXT, created_at TEXT, updated_at TEXT)
messages(id INTEGER PK, conversation_id TEXT FK, role TEXT, content TEXT, thinking TEXT, tool_calls TEXT, tool_name TEXT, created_at TEXT)
```

API:
- `create_conversation(title, model) → id`
- `list_conversations(limit, offset) → [...]`
- `get_messages(conversation_id) → [...]`
- `add_message(conversation_id, role, content, thinking, tool_calls, tool_name)`
- `delete_conversation(conversation_id)`
- `update_title(conversation_id, title)`
- `search_conversations(query) → [...]`

2.2 **集成到 `core/agent.py`**
- `Agent.run()` / `Agent.stream()` 接受 `conversation_id` 参数
- 每轮工具调用和回复自动持久化到 SQLite
- 加载历史消息时从 SQLite 读取而非依赖客户端发送完整历史

**依赖**：Phase 1 完成后

## Phase 3: TUI 应用 (Textual)

> 构建终端交互界面，实现用户描述的多面板布局。

### 3.1 项目结构

```
tui/
├── __init__.py
├── __main__.py              # python -m tui 入口
├── app.py                   # AgentApp(textual.App) 主应用
├── screens/
│   └── chat_screen.py       # ChatScreen — 主界面 Screen
├── widgets/
│   ├── conversation_list.py # 左侧栏：对话列表 (ListView)
│   ├── chat_view.py         # 主区域：消息流 (VerticalScroll)
│   ├── message_widget.py    # 单条消息：用户/AI/工具状态
│   ├── tool_call_widget.py  # 可折叠工具调用详情 (Collapsible)
│   ├── thinking_widget.py   # 可折叠思考过程 (Collapsible)
│   ├── file_explorer.py     # 底部面板：workspace 文件树 (Tree)
│   └── input_bar.py         # 输入栏：TextArea + 文件/URL 检测
├── styles/
│   └── app.tcss             # Textual CSS 样式
└── utils.py                 # 辅助函数（路径美化、URL 检测等）
```

### 3.2 布局设计 (Textual CSS)

```
Screen
├── Horizontal (flex)
│   ├── ConversationList (width: 25%, min-width: 20)
│   └── Vertical (flex: 1)
│       ├── ChatView (flex: 3, scrollable)
│       ├── InputBar (height: auto, min-height: 3)
│       └── FileExplorer (height: 25%, collapsible)
```

### 步骤

3.3 **ConversationList 组件** — 左侧栏
- 从 SQLite 加载对话列表（标题、时间、消息数）
- 点击切换对话 → 重新加载 ChatView
- 快捷键：`n` 新建对话、`d` 删除对话
- 格式：`标题 (2h ago, 24 msgs)`

3.4 **ChatView 组件** — 主聊天区域
- VerticalScroll 容器，包含 MessageWidget 列表
- **UserMessage**：右对齐或带 `>` 前缀，检测并高亮 `【文件名.ext】` 和 `[URL]`
- **AssistantMessage**：Markdown 渲染（Rich），左对齐
- **ThinkingWidget**：Collapsible，默认折叠，标题显示 "💭 思考过程"
- **ToolCallWidget**：Collapsible，默认折叠，标题显示 "🔧 tool_name → 参数摘要 — ✅/❌"
- 流式更新：async worker 消费 `Agent.stream()` 的 token，实时追加到当前 AssistantMessage

3.5 **InputBar 组件**
- TextArea，支持多行输入（Shift+Enter 换行，Enter 发送）
- **文件路径检测**：当用户拖入文件（终端粘贴绝对路径），正则检测 Windows/Unix 路径
  - 渲染为 `【filename.ext】` 视觉框
  - 发送时自动预取文件内容（复用 `Agent.prefetch_files()`）
- **URL 检测**：正则检测 `http(s)://...`，渲染为 `🔗 domain.com/path`
- 提示文字：`输入消息... (拖入文件或粘贴 URL)`

3.6 **FileExplorer 组件** — 底部面板
- Textual `Tree` 组件，根节点为 workspace 目录 (`data/workspace/`)
- 懒加载：点击三角展开子目录时才读取（直接读本地文件系统，不经过 skill-files）
- 显示：文件名 + 大小 + 类型图标
- 双击文件 → 将路径插入 InputBar
- 快捷键：`Ctrl+E` 切换显示/隐藏

3.7 **流式渲染集成**
- 使用 Textual 的 `Worker` (async) 运行 `Agent.stream()`
- token 到达时通过 `call_from_thread` / `post_message` 更新 ChatView
- 工具执行期间显示 spinner + 工具名
- 完成后更新工具状态为 ✅/❌

3.8 **主题样式** (*parallel with 3.3-3.7*)
- 深色终端主题（类似截图中的黑底绿字风格）
- 边框分隔各面板
- 标题栏显示 `Local AI Agent v2.0`
- 颜色方案：深灰背景、白色文字、绿色高亮、蓝色链接

**依赖**：Phase 1 + Phase 2 完成后；3.3-3.6 可 parallel 开发，3.7 依赖 3.4+3.5

## 验证步骤

1. **Core 单元测试** — 直接调用 `Agent.run()` 传入测试消息，验证 agentic loop 正常工作（无需 Docker skill 服务可 mock tool_router）
2. **SQLite 验证** — 创建对话 → 添加消息 → 列出对话 → 加载历史 → 删除对话
3. **Docker skill 连通性** — `docker compose up -d` 后，从 host 直接 `curl http://localhost:8100/health` 和 `http://localhost:8200/health` 验证端口映射
4. **TUI 启动** — `python -m tui` 启动，验证布局渲染（对话列表、聊天区、文件浏览器）
5. **端到端对话** — 在 TUI 中发送消息 → 流式响应 → 工具调用（file_read/code_exec）→ 对话保存到 SQLite → 切换对话 → 重新加载历史
6. **文件拖入** — 拖入文件路径到 InputBar → 检测并渲染为 `【filename】` → 发送后预取内容
7. **文件浏览器** — 展开 workspace 目录 → 双击文件 → 路径插入输入栏
8. **记忆写回** — 对话后检查 `/workspace/.memory/MEMORY.md` 和 `conversations/{id}.md` 更新
9. **启动脚本** — `scripts/quick-start.ps1` 完整流程验证

## 决策记录

- **Textual 框架**：原生支持多面板 CSS 布局、Tree/ListView 组件、异步 Worker，完美匹配需求
- **Gateway 嵌入 TUI**：TUI 直连 Ollama (localhost:11434) + Docker skill 服务，去除中间 HTTP 层，降低延迟
- **SQLite 对话存储**：支持搜索/分页/快速查询，适合长期使用
- **不保留 Open WebUI 兼容**：彻底移除 OpenAI-compatible API，简化架构
- **移除 Tailscale 远程访问**：用户明确要求
- **保留 skill CRUD 工具**：6 个工具（register/unregister/info/update/list/run）为 AI agent 服务，无需简化
- **保留 Docker skill 沙箱**：skill-files 和 skill-runner 的 Docker 容器隔离是安全基础，不可去除
- **Conda 不再需要**：Open WebUI 移除后，TUI 直接 pip install 即可

## 范围边界

**包含**：TUI 前端、Core 提取、SQLite 存储、Docker 简化、记忆统一、脚本重写、远程访问移除
**排除**：skill 服务内部代码不改动、现有转换器插件不改动、config/tools/*.yaml 工具定义不改动、policy.yaml 不改动
