# Claude Code 源码分析 — 对 local-ai-agent 框架的借鉴方案

## 一、两个框架的对比总览

| 维度 | local-ai-agent (当前) | Claude Code (参考) |
|------|----------------------|-------------------|
| **架构** | 微服务 (Gateway + 3 Skill Services) | 单体 CLI + 子进程 fork |
| **LLM** | 本地 Ollama (小模型 7B-32B) | Claude API (大模型) |
| **Context 管理** | 无 — 完全依赖客户端 (Open WebUI) | 多层压缩: MicroCompact → AutoCompact → SessionMemory |
| **Memory** | 无持久记忆 | 4 类型文件记忆 + 会话笔记 + 团队同步 + 背景自动提取 |
| **Tool 注册** | YAML 声明式 (19 个工具) | TypeScript 类 + buildTool() 工厂 + MCP 协议 (40+ 工具) |
| **Tool 调度** | 线性: Policy → Route → Execute | 分层: Rules → Hooks → Classifier → Execute → Post-Hooks |
| **Prompt** | 单一 system.txt 静态文件 | 模块化分段组装, 静态/动态分界, 按角色/场景切换 |
| **Sub-Agent** | 无 | AgentTool fork 子上下文, 背景任务, 团队协作 |
| **Skill** | Python 动态注册 (workspace/skills/) | Markdown frontmatter + 多来源发现 (本地/bundled/MCP/plugin) |

## 二、核心借鉴领域分析

### 领域 A: Context 利用效率

**问题**: 本地小模型 context window 通常只有 4K-32K tokens, 比 Claude 的 200K 小得多, context 管理更加关键。

**Claude Code 的做法**:
1. **三层压缩机制**:
   - **MicroCompact**: 清除旧 tool 调用结果 (file_read/shell 输出), 替换为 `[Old tool result cleared]`
   - **AutoCompact**: 超过阈值时, fork 一个子 agent 对整个对话做 9 段式结构化摘要
   - **SessionMemory**: 将摘要持久化, 下次恢复时直接加载
2. **Token 预算追踪**: 实时统计 token 用量, 90% 阈值自动触发压缩, 递减收益检测
3. **Memoization 缓存**: system context / git status 等只计算一次, 跨 turn 复用
4. **Git status 截断**: 2000 字符上限, 防止大仓库状态撑爆 context
5. **API-round 分组**: 按 assistant 响应边界分组, 安全截断单位

**对 local-ai-agent 的建议**:

| 编号 | 改进项 | 优先级 | 说明 |
|------|--------|--------|------|
| A1 | **实现 Tool Result 压缩 (MicroCompact)** | 🔴 高 | 小模型 context 极其宝贵, file_read/code_exec 的大段输出应在后续 turn 替换为摘要。可在 gateway 的消息历史管理中实现。 |
| A2 | **实现对话自动摘要 (AutoCompact)** | 🔴 高 | 当消息历史超过 token 阈值时, 调用 LLM 生成结构化摘要替换旧消息。对小模型尤其重要 — 建议阈值设为 context_window × 60%。 |
| A3 | **Token 预算追踪器** | 🟡 中 | 在 gateway 中添加 token 计数 (可用 tiktoken 或字节估算), 实时追踪每轮消耗, 触发压缩决策。 |
| A4 | **System Context 缓存** | 🟡 中 | 将 system prompt + 工具描述等静态内容缓存, 避免每次请求重新拼装。 |
| A5 | **Tool 输出截断策略** | 🟡 中 | 为每种 tool 设定 max_result_size_chars (如 file_read: 8000, code_exec: 4000), 超出部分截断并标注 `[truncated, showing first N chars]`。 |

### 领域 B: Memory 管理

**问题**: 当前框架完全无记忆, 每次对话从零开始, 对重复性任务 (如数据分析流程) 效率低下。

**Claude Code 的做法**:
1. **4 种记忆类型**: User (个人偏好) / Feedback (反馈修正) / Project (项目状态) / Reference (外部链接)
2. **背景自动提取**: 主对话结束一轮后, fork 子 agent 自动从对话中提取有价值的记忆
3. **语义召回**: 用 Sonnet 模型对记忆库做语义匹配, 选出最相关的 5 条注入 prompt
4. **过期警告**: 超过 1 天的记忆附加 freshness note, 防止过时信息被当作事实
5. **会话笔记**: 10 段式模板 (标题/当前状态/任务规格/文件函数/工作流/错误修正/...)
6. **MEMORY.md 索引**: 最多 200 行的目录文件, 始终注入 system prompt

**对 local-ai-agent 的建议**:

| 编号 | 改进项 | 优先级 | 说明 |
|------|--------|--------|------|
| B1 | **实现 Workspace Memory 系统** | 🔴 高 | 在 `/workspace/.memory/` 下维护 Markdown 记忆文件 + MEMORY.md 索引。记忆类型可简化为: `user` (用户偏好) / `project` (项目笔记) / `reference` (常用链接/命令)。 |
| B2 | **对话结束自动提取** | 🟡 中 | 在 tool_loop 结束时, 额外调用一次 LLM 提取本轮对话的关键信息写入记忆。小模型可用精简 prompt 降低成本。 |
| B3 | **记忆注入 System Prompt** | 🔴 高 | 每次对话开始时, 读取 MEMORY.md 索引注入 system prompt, 让模型了解 workspace 的历史上下文。 |
| B4 | **简单的关键词召回** | 🟡 中 | 大模型可用语义匹配, 小模型建议用关键词/TF-IDF 匹配从记忆库选出相关条目, 降低计算成本。 |
| B5 | **记忆过期标注** | 🟢 低 | 对超过 N 天未更新的记忆, 注入时附加 `[此记忆已过期 X 天, 请核实]` 提示。 |

### 领域 C: Tool / Skill 调用优化

**问题**: 当前 tool 系统功能完备但缺少高级调度特性, 小模型容易在 tool 选择上犯错。

**Claude Code 的做法**:
1. **Tool Deference (延迟加载)**: 非核心 tool 只发送名称+简短提示, 模型需要时通过 ToolSearch 查找全量描述
2. **buildTool() 工厂模式**: 统一的 tool 接口 (isReadOnly, isDestructive, isConcurrencySafe, checkPermissions)
3. **Pre/Post Hooks**: 执行前后的扩展点, 可拦截/修改/审计
4. **Sub-Agent 分工**: AgentTool fork 独立上下文执行复杂子任务, 结果返回主对话
5. **Skill Frontmatter**: Markdown 文件 + YAML 头部定义 skill 元数据 (描述/何时使用/允许的工具/模型)
6. **结果大小控制**: maxResultSizeChars, 超大结果写磁盘只返回摘要

**对 local-ai-agent 的建议**:

| 编号 | 改进项 | 优先级 | 说明 |
|------|--------|--------|------|
| C1 | **Tool 分级加载** | 🔴 高 | 对小模型, 19 个 tool 的完整描述可能占用过多 context。建议分为 "核心 tool" (file_read/write/code_exec, 始终加载) 和 "扩展 tool" (git/web/skill_*, 按需描述)。 |
| C2 | **Tool 描述精简** | 🔴 高 | 为小模型优化 YAML 中的 description, 用更短更精准的措辞。Claude Code 的 searchHint (3-10 词) 可作参考。 |
| C3 | **Tool 调用结果模板化** | 🟡 中 | 统一 tool 返回格式 (status/data/error), 让小模型更容易解析。当前各 skill 返回格式不一致。 |
| C4 | **增加 tool 元数据** | 🟡 中 | 在 YAML 中增加 `is_read_only`, `is_destructive`, `max_result_chars` 字段, gateway 可据此做智能决策。 |
| C5 | **Skill 发现增强** | 🟢 低 | 将已注册 skill 的摘要注入 system prompt, 让模型知道有哪些 skill 可用, 而不需要先调用 skill_list。 |

### 领域 D: Prompt 工程 (通用场景: 文件分析/文本总结/数据提炼)

**问题**: 框架不只用于编程, 还需支持文档分析、数据处理、文本总结等任务。当前 system prompt 偏向编程操作。

**Claude Code 的 Prompt 工程精华**:

1. **模块化分段组装**:
   - system prompt 是一个 string 数组, 每段可独立启用/禁用
   - 静态/动态分界标记, 优化缓存
   - 按场景 (coordinator/agent/plan/custom) 切换不同 prompt 组合

2. **结构化摘要 prompt (9 段式)**:
   ```
   1. 主要请求与意图
   2. 关键技术概念
   3. 文件和代码段
   4. 错误和修复
   5. 问题解决过程
   6. 所有用户消息 (非 tool 结果)
   7. 待办任务
   8. 当前工作 (最近 2-3 轮 + 代码片段)
   9. 可选的下一步 (含直接引用)
   ```

3. **反漂移机制**: 摘要中保留原文直接引用, 防止多轮压缩后信息失真

4. **Magic Docs 自动更新**: 后台 agent 自动更新项目文档, 强调 "WHY > HOW > WHAT"

5. **Session Memory 模板**: 10 段式笔记模板, 带 token 预算 (每段 ~2000 tokens)

6. **数值化长度锚点**: 用精确 token 数约束输出长度, 比 "简洁" 更有效

**对 local-ai-agent 的建议**:

| 编号 | 改进项 | 优先级 | 说明 |
|------|--------|--------|------|
| D1 | **模块化 System Prompt** | 🔴 高 | 将 system.txt 拆分为多个模块: `base.txt` (身份/语言/风格) + `tools.txt` (工具指导) + `analysis.txt` (数据分析) + `summarize.txt` (文本总结) + `workspace.txt` (workspace 信息)。根据用户意图动态组装。 |
| D2 | **场景化 Prompt 模板** | 🔴 高 | 为不同任务场景预设 prompt 模板: <br>- **数据分析**: 引导 "先 file_list → 再 file_read 采样 → 再 code_exec 分析 → 最后 file_write 报告" <br>- **文本总结**: 引导 "读取 → 提炼要点 → 结构化输出" <br>- **文件对比**: 引导 "分别读取 → 对比关键差异 → 输出表格" |
| D3 | **结构化摘要 Prompt** | 🟡 中 | 为 AutoCompact 设计适合小模型的摘要 prompt, 可简化为 5 段: 用户意图 / 关键数据 / 已完成步骤 / 当前状态 / 下一步。 |
| D4 | **输出格式引导** | 🟡 中 | 在 system prompt 中为小模型提供明确的输出格式示例 (表格/列表/摘要), 用数值锚点约束长度 (如 "摘要不超过 200 字")。 |
| D5 | **Workspace 上下文注入** | 🟡 中 | 对话开始时自动注入 workspace 概况 (file_list 结果摘要 + 记忆索引), 减少模型 "盲猜" 文件名。类似 Claude Code 的 git status + claude.md 注入。 |
| D6 | **意图检测 + Prompt 路由** | 🟢 低 | 在 gateway 中增加轻量意图分类 (关键词匹配或小模型推理): 编程/数据分析/文本总结/通用问答, 然后路由到对应 prompt 模板。 |

### 领域 E: 架构层面的借鉴

| 编号 | 改进项 | 优先级 | 说明 |

## 三、实施路线图 (按优先级分阶段)

### 第一阶段: Context 效率 + Prompt 模块化 (最高 ROI)

- ✅ **E1** Gateway 端消息历史管理 — 新增 `context_manager.py` (MicroCompact + token 估算) + `session_store.py` (指纹会话存储, TTL 2h); 集成到 streaming/non-streaming 双路径; 环境变量 `CONTEXT_WINDOW` / `COMPACT_THRESHOLD` 可配

**Batch 3: B3 — 记忆注入 System Prompt (依赖 B1)**

- **B3** 记忆注入 System Prompt

  - 注入 system prompt 尾部, 让模型了解 workspace 历史
  - 与 D5 的 workspace 概况一起注入, 共享 extra_sections 机制

## 四、关键设计决策备忘

### 小模型 vs 大模型的差异化考量

| 特性 | Claude Code (大模型) | local-ai-agent (小模型) 建议 |
|------|---------------------|----------------------------|
| 摘要 | fork 子 agent 用同一大模型做摘要 | 可复用同一小模型, 但需要更简短的摘要 prompt |
| 记忆召回 | Sonnet 语义匹配选 top-5 | 用关键词/TF-IDF 匹配, 成本更低 |
| Tool Deference | 模型自主搜索需要的 tool | 小模型难以可靠做 tool search, 建议 gateway 端基于意图预选 tool 子集 |
| Prompt 长度 | system prompt 可占 10K+ tokens | 需严格控制在 2K-4K tokens, 每个模块尽可能精简 |
| 输出控制 | 数值锚点 + 质性指令 | 小模型更依赖数值锚点和格式示例 |
| 并行 tool 调用 | 模型自主决定并行 | 小模型一般不支持, 由 gateway 按依赖关系编排 |

### 实现要点

1. **E1 (Gateway 消息管理) 是第一阶段的基础** — 没有服务端消息管理, 就无法实现 MicroCompact 和 AutoCompact
2. **Token 估算可简化** — 对小模型 (LLaMA/Qwen), 中文约 1.5 字/token, 英文约 4 字符/token, 粗估即可
3. **摘要 Prompt 要极度精简** — 小模型的 instruction following 能力有限, 摘要 prompt 不宜超过 500 字
4. **Memory 系统先 MVP** — 先实现手动读写 + system prompt 注入, 再加自动提取
